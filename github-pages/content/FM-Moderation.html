Introduction

The Infosys Responsible AI Toolkit, a technical offering within the Shield component of the 3S (Scan, Shield, Steer) framework, provides solutions for detecting and addressing risks to ensure responsible AI.

The Infosys Responsible AI toolkit features a suite of APIs, known as a FM-Moderation Layer, to safeguard AI models by verifying inputs and outputs for safety, security, privacy, and fairness. Additionally, These APIs assist in providing explanations for the outcomes of various traditional and generative AI models, ensuring transparency in their decision-making processes.

The moderation layer is a modular component that can be seamlessly integrated into clients’ AI systems. It enables integration with custom traditional and large language models, implementation of security measures, configuration of moderation checks, and ongoing monitoring of AI models to maintain ethical, legal, and compliance standards.

Infosys Responsible AI toolkit: FM - Moderation Layer

The Filtering and Moderation module (called as FM-Moderation layer) is a crucial part in content moderation systems that ensure the safety, compliance, and appropriateness of responses generated by AI models like GPT3.5 and GPT4. It applies a set of checks and filters to both the input prompt and the generated response to keep control over the content generated by the language model. The module consists of multiple layers, including Request Moderation, Response Moderation, and Response Comparison, each serving a specific purpose in the moderation process.

Infosys Responsible AI toolkit offers a user-friendly interface (UI) and a suite of APIs to facilitate the moderation of inputs and outputs for generative and traditional AI models. This section explores how to perform moderation using the interface and provides in-depth details about the related APIs. The key specifications and deployment steps for different cloud platforms, including Azure, GCP, and AWS are outlined in this section.

5.1 Invisible Text Check (Package-based evaluation)

  • The Invisible Text Scanner is designed to detect and remove non-printable, invisible Unicode characters from text inputs.

  • This is crucial for maintaining text integrity in Large Language Models (LLMs) and safeguarding against steganography-based attacks.

  • We are using “unicodedata” python package to detect invisible characters.

  • The scanner targets invisible Unicode characters, particularly in the Private Use Areas (PUA) of Unicode, which include:

  •      'Cf':   Format characters

  •      'Co':  Private use characters

  •      'Cn':  Unassigned characters

  •      ‘So’:  Other symbols (ex: emojis)

  •      ‘Sc’:   Currency symbols

5.2 Gibberish Check (Model-based evaluation) –

  • This scanner is designed to identify and filter out gibberish or nonsensical inputs in English language text.

  • Gibberish refers to nonsensical or meaningless language that lacks any discernible meaning, identified by use of random words, nonsensical phrases, grammatical errors, or syntactical abnormalities.

We have used Hugging Face pretrained model for this check: https://huggingface.co/madhurjindal/autonlp-Gibberish-Detector-492513457

  • We are dealing with 4 types of categories :

  1. Noise: Gibberish at the zero level where even the different constituents of the input phrase (words) do not hold any meaning independently.
    Example: dfdfer fgerfow2e0d qsqskdsd djksdnfkff swq.

  2. Word Salad: Gibberish at level 1 where words make sense independently, but when looked at the bigger picture (the phrase) any meaning is not depicted.
    Example: 22 madhur old punjab pickle chennai.

  3. Mild gibberish: Gibberish at level 2 where there is a part of the sentence that has grammatical errors, word sense errors, or any syntactical abnormalities, which leads the sentence to miss out on a coherent meaning.
    Example: Madhur study in a teacher.

  4. Clean: This category represents a set of words that form a complete and meaningful sentence on its own.
    Example: I love this website.

FM - Moderation through UI

The following section illustrates the application of the FM-moderation module from the Infosys Responsible AI toolkit to regulate the input and output of an AI model. The explanation is divided into two parts, where the first part focuses on template-based guardrails and the second part discusses model-based guardrails.

Moderation through Template based guardrails

  1. Navigate to the designated application window and find the section about generative AI models. From the options presented on the left-hand side of the interface, select the module labeled 'FM Moderation’.

  1. In FM moderation with multilingual support, prompts can be sent in any language. If the prompt is not in English, users have the option to translate it using the google translate icon. Following translation, users select from three options: Azure, Google, or none. Internally, prompts are automatically converted to English for backend processing.

  1. Additionally, prompts may include emojis when emoji support is enabled in the prompt field. This ensures that the model considers emojis when generating responses. This approach eases seamless moderation across languages and incorporates emoji interpretation where applicable, enhancing the overall effectiveness and user experience of the moderation process.

  1. To ensure precise and effective execution of the task, please select "FM moderation" from the options on the left-hand side, set the temperature value to 0, and choose the use case as "demo" to ensure consistency and accuracy. The prompt template should be crafted to prioritize goal achievement while simplifying complexity. Employ the GPT-4 and Nemo Guardrail models to streamline this process effectively. Additionally, include comparisons related to hallucination and uncertainty to enhance the accuracy of the generated output. This approach will support achieving the desired results while minimizing potential errors.

The output panel have three layers such as request moderation, response moderation, response comparison. Each layer is having various checks.
